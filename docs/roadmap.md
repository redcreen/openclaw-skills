# Roadmap

| Stage | Status | Goal | Exit Criteria | Unlocks |
| --- | --- | --- | --- | --- |
| [Stage 1](reference/openclaw-skills/development-plan.md#stage-1-workspace-governance-baseline) | done | Define the workspace as a multi-skill-set repository with explicit isolation and indexing rules | Root docs, architecture, and control surface agree on the top-level module boundary | Consistent future module onboarding |
| [Stage 2](reference/openclaw-skills/development-plan.md#stage-2-health-skill-set-baseline) | done | Establish `health/` as the first isolated skill set with independent skill folders and README surfaces | `health` landing docs and first skill skeletons exist | Functional health skill implementation |
| [Stage 3](reference/openclaw-skills/development-plan.md#stage-3-health-archive-functional-delivery) | done | Deliver `health-archive` as a local-first archive skill with explicit success status | Raw evidence backup, normalized record write, and status reporting work end to end | Trustworthy health intake |
| [Stage 4](reference/openclaw-skills/development-plan.md#stage-4-private-doctor-functional-delivery) | done | Deliver `private-doctor` as a concise family-doctor skill using the same local health contract | Stable interpretation, planning, and follow-up behavior exist | Usable health doctor workflow |
| [Stage 5](reference/openclaw-skills/development-plan.md#stage-5-future-skill-set-expansion-and-guardrails) | active | Add future skill sets such as `order/` without re-opening the structure debate, and give each skill a GitHub direct-install entry | New modules follow the same install, doc, isolation, and release contracts | Repeatable multi-domain growth |
